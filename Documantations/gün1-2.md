## 1. Çalışmanın Amacı

Bu çalışmanın amacı JoedTech platformuna AI agent entegrasyonun mümkün olup olmadığını, eğer mümkünse hangi teknik altyapı gereksinimlerine ihtiyaç duyulduğunu araştırmaktır.

## 2. Mevcut Sistem Yapısı Analizi

JoedTech platformu:

- Multi-tenant (marka / şube bazlı)  
- Feature-based yetkilendirme  
- Rol bazlı erişim (RBAC)  

Backend tarafında zorunlu kurallar içeriyor:

- Kullanıcı kendini silemez  
- Kendi rolünü güncelleyemez  
- Sistemde en az 1 admin kalmalıdır  
- Yetki enforcement backend tarafında yapılmaktadır  

Bu yapı, AI entegrasyonu için sağlam bir güvenlik temeli sunmaktadır.

---

## 3. AI Agent Konseptinin Değerlendirilmesi

### 3.1 AI’nın Sistemde Konumlandırılması

AI’nın UI üzerinden çalışan bir bot değil, backend üzerinden işlem yapan bir service account (AI_SYSTEM user type) olarak konumlandırılması gerektiği sonucuna varılmıştır.

**Planlanan yapı:**

- `user_type: AI_SYSTEM`  
- Branch bazlı scope  
- Feature bazlı permission  
- UI login kapalı  
- Sadece API üzerinden erişim  

Bu sayede AI, sistem içinde izlenebilir ve kontrol edilebilir bir kullanıcı olur.

## 4. Teknik Mimari Değerlendirmesi

AI doğrudan veritabanına veya ham API’lere erişmemelidir.

**Önerilen akış:**

User / Event
↓
LLM
↓
Structured Tool Call
↓
Validation Layer
↓
Business Rule Engine
↓
REST API
↓
Database


Bu mimaride:

- Backend kuralları korunur  
- Yetki escalation engellenir  
- Self-modification mümkün olmaz  
- Audit log tutulur  

**Not:**  
MCP (Model Context Protocol) bu aşamada gerekli görülmemiştir.  
OpenAI function calling + mevcut REST API altyapısı yeterlidir.

## 6. Veri Güvenliği Değerlendirmesi

JoedTech sistemi tüm kişisel verileri içermektedir:

- Kimlik bilgileri  
- İletişim bilgileri  
- Finansal bilgiler  
- Eğitim geçmişi  

Bu nedenle:

AI’ya ham PII (Personally Identifiable Information) gönderilmemelidir.

**Uygulanması gereken prensip:**

👉 Data Minimization + Tokenization  

**AI’ya gönderilecek veri:**

- internal_id  
- metrikler  
- sayısal özetler  
- davranış skorları  

AI karar üretir, gerçek kimlik bilgileri backend’de kalır.

Bu yapı KVKK açısından daha güvenli bir model sunar.

## 7. Risk Analizi !!! Tekrar Değerlendirilecek !!!

Tespit edilen başlıca riskler:

1. Prompt injection  
2. Yetki escalation  
3. Toplu yanlış işlem (bulk destructive actions)  
4. Yurt dışına veri aktarımı (SaaS LLM kullanımı halinde)  
5. Token çalınması?  
6. Audit yoksa hukuki risk?  

**Risk azaltma yöntemleri:**

- Backend hard constraints  
- Validation layer  
- Feature-scoped AI  
- Branch-level AI isolation  
- Audit logging  
- PII masking
